# ADI icon workflow parity — design spec

- **Date:** 2026-04-19
- **Status:** approved for planning
- **Sub-project 1 of 2.** Sub-project 2 (extending both libraries with new blocks) gets its own brainstorm/spec cycle after this lands.

## Goal

Bring the ADI component library up to parity with the SW library's source-file workflow so icons can be modified, added, and diffed without hand-editing base64 blobs. Replace the missing `embed-icons.sh` (referenced only in a stale header comment) with a Python generator that also handles SW and keeps the `ADI_COMPONENTS` / `SW_COMPONENTS` Python lists in `d2/__init__.py` in sync. Add CI drift detection.

## Non-goals

- Adding new components (Sub-project 2).
- Changing icon visual design or theme files.
- Altering the Go-side compilation pipeline. The Go library embeds `lib/{adi,sw}/*-components.d2` verbatim via `//go:embed`; this spec only changes *how those files get produced*, not what ends up in them.

## Design decisions

| Decision | Choice |
|---|---|
| Decomposition | Workflow parity first (this spec); new blocks later (Sub-project 2) |
| Generator language | Python (one `scripts/embed_icons.py` handles both libraries) |
| Retire `embed-sw-icons.sh` | Yes — deleted when the Python generator lands |
| Extraction of existing ADI blobs | Committed one-shot script `scripts/extract_adi_icons.py` |
| `ADI_COMPONENTS` / `SW_COMPONENTS` lists in `d2/__init__.py` | Regenerated by `embed_icons.py` between `# BEGIN AUTOGEN:<lib>` / `# END AUTOGEN:<lib>` markers |
| Drift detection | `nox -s embed_check` runs `embed_icons.py --check`, wired into CI |

## File layout

### New files

- `scripts/extract_adi_icons.py` — one-shot migration. Parses `lib/adi/adi-components.d2`, decodes each base64 blob, writes each to `lib/adi/icons/<name>.svg`. Idempotent; self-verifying (runs the regenerator in `--check` mode before exiting).
- `scripts/embed_icons.py` — ongoing generator + drift checker. CLI: `python scripts/embed_icons.py --library {adi,sw,all} [--check]`.
- `lib/adi/icons/*.svg` — 64 files with names matching the existing class names (`adc.svg`, `dac.svg`, `amplifier.svg`, `filter-lp.svg`, …).
- `test/test_embed_icons.py` — unit tests for the generator.

### Modified files

- `lib/adi/adi-components.d2` — regenerated. Header rewritten to:
  ```
  # Auto-generated by scripts/embed_icons.py — do not edit manually
  # Regenerate with: python scripts/embed_icons.py --library adi
  ```
  Body must be byte-identical to the current file post-header-edit.
- `lib/sw/sw-components.d2` — regenerated under the same Python generator. Header updated analogously. Any whitespace shift between the bash-produced output and the Python-produced output is landed in the same commit; the Go library rebuild + existing test suite prove functional equivalence.
- `d2/__init__.py` — `ADI_COMPONENTS` and `SW_COMPONENTS` list literals are wrapped with `# BEGIN AUTOGEN:<lib>` / `# END AUTOGEN:<lib>` markers. The generator only rewrites content between those markers. `ADI_THEME_CLASSES`, `SW_THEME_CLASSES`, and every other symbol in the file are untouched.
- `noxfile.py` — new `embed_check` session (see below).
- `.github/workflows/build_wheels.yml` (or the test workflow) — add the new nox session so every PR runs it.
- `CLAUDE.md` — update the "SW components" bullet to reference `embed_icons.py` instead of `embed-sw-icons.sh`; add a parallel "ADI components" bullet.

### Deleted

- `scripts/embed-sw-icons.sh`.

## Generator contract

### CLI

```
scripts/embed_icons.py --library {adi,sw,all} [--check]
```

- `--library adi` / `--library sw` — act on one library.
- `--library all` — act on both, in order `adi`, `sw`. Useful for CI check and for regenerating everything after an update.
- `--check` — do not write any files. Generate into memory and compare against the checked-in files. Exit 0 if identical, 1 with a unified diff on stderr if different.

### Output format for `lib/<lib>/<lib>-components.d2`

```
# Auto-generated by scripts/embed_icons.py — do not edit manually
# Regenerate with: python scripts/embed_icons.py --library <lib>

classes: {
  <name-1>: {
    icon: "data:image/svg+xml;base64,<b64>"
  }
  <name-2>: {
    icon: "data:image/svg+xml;base64,<b64>"
  }
  ...
}
```

- Two-space indent, no trailing whitespace.
- Each `<b64>` is unwrapped (no embedded newlines) — equivalent to `base64 -w0` output.
- Ordering follows a script-local `_ORDER` list keyed by library (see next section).

### Ordering

`ADI_COMPONENTS` in today's `d2/__init__.py` has a non-alphabetical domain-grouped ordering (`adc`, `dac`, `amplifier`, `filter-lp`, …). Sorting alphabetically would churn both the `.d2` file and the Python list for no functional reason. The generator preserves the current order by reading it from a dict at the top of `scripts/embed_icons.py`:

```python
_ORDER = {
    "adi": ["adc", "dac", "amplifier", "filter-lp", ..., "digipot"],  # 64 entries, copied from current ADI_COMPONENTS
    "sw": sorted(icon filenames under lib/sw/icons/),                  # 32 entries, alphabetical
}
```

`_ORDER["sw"]` is derived at script startup from the filesystem (since `SW_COMPONENTS` is already alphabetical). `_ORDER["adi"]` is a literal list seeded from the current `ADI_COMPONENTS` contents. Sub-project 2 will explicitly append new entries to `_ORDER["adi"]` when new blocks are added.

### Validation inside the generator

For each library:
- Every entry in `_ORDER[lib]` must have a corresponding `lib/<lib>/icons/<name>.svg` file. Missing source → exit with `missing icon: lib/<lib>/icons/<name>.svg`.
- Every `.svg` file in `lib/<lib>/icons/` must appear in `_ORDER[lib]`. Orphan → exit with `unknown icon: <name>.svg — add it to _ORDER['<lib>']`.

Both conditions are verified before writing or diffing.

### `d2/__init__.py` list regeneration

For each library, the generator locates the region between `# BEGIN AUTOGEN:<lib>` and `# END AUTOGEN:<lib>` and replaces it with:

```python
# BEGIN AUTOGEN:<lib> — regenerated by scripts/embed_icons.py; do not edit by hand
<LIB>_COMPONENTS = [
    "<name-1>",
    "<name-2>",
    ...
]
# END AUTOGEN:<lib>
```

The initial commit lays down the markers in place. Thereafter only the interior of each marker block changes.

## Extraction script (`scripts/extract_adi_icons.py`)

### Purpose

One-shot migration. Produces `lib/adi/icons/*.svg` from the 64 base64 blobs in the current `lib/adi/adi-components.d2`.

### Flow

1. Read `lib/adi/adi-components.d2`.
2. For each `<name>: { icon: "data:image/svg+xml;base64,<b64>" }` block, `base64.b64decode(<b64>)` and write to `lib/adi/icons/<name>.svg`.
3. Run `scripts/embed_icons.py --library adi --check` in-process.
4. On check-mode diff: abort with a clear error and a reminder that the extraction produced SVGs that don't round-trip. User needs to inspect. (This indicates a bug in either script.)
5. On check-mode success: print `Extracted 64 icons to lib/adi/icons/. Regen round-trips: OK.`

### Self-verification

The in-process check after extraction is the load-bearing guarantee that nothing was lost in translation. If the migration commit passes CI (which runs `embed_check`), the extraction is proven lossless by construction.

## Drift check (CI)

### `nox -s embed_check` session

```python
@nox.session
def embed_check(session: nox.Session) -> None:
    """Verify *-components.d2 and d2/__init__.py lists match the icon sources."""
    session.run("python", "scripts/embed_icons.py", "--library", "all", "--check", external=True)
```

No `session.install(...)` call — the generator depends only on Python stdlib.

### CI wiring

Add `embed_check` to the existing `.github/workflows/` test matrix. Runtime is in single-digit seconds (96 base64 encodes, a couple of regex rewrites, a unified-diff on text that's already in memory).

### Check-mode behaviour

- Exit 0 on sync.
- Exit 1 on drift. stderr receives a unified diff showing exactly which icon or list entry differs. Goal: a contributor who forgot to regen sees the diff, runs `python scripts/embed_icons.py --library all`, commits, pushes.
- No writes in check mode — strictly read-only. Check mode is safe to run anywhere.

## Testing

`test/test_embed_icons.py` — new, small unit test file.

| Test | What it asserts |
|---|---|
| `test_generator_is_deterministic` | Running the generator twice on an unchanged tree produces identical output both times. |
| `test_check_mode_detects_modified_icon` | Copy `lib/adi/icons/` to `tmp_path`, mutate one SVG byte, run the generator's core function with the `tmp_path` as the icons dir, assert check-mode exit-nonzero and a diff that names the icon. |
| `test_check_mode_detects_missing_icon` | Same harness; delete one SVG; assert failure with `missing icon: ... <name>.svg`. |
| `test_orphan_icon_fails_loudly` | Same harness; add an extra `zzz-new.svg`; assert refusal with `unknown icon: zzz-new.svg — add it to _ORDER['adi']`. |

Implementation note: the generator's core logic is factored into a function that accepts paths, so tests can point at `tmp_path` without mutating the repo. The CLI is a thin shell over that function.

No new tests are needed for the existing compile / directive / cache paths. The Go library embeds the `.d2` files at `go build` time; if the extraction round-trips and the SW regen is functionally identical, the rebuilt shared library is byte-identical and the existing `test/test_d2.py` + `test/test_sphinx_*.py` suites already prove end-to-end correctness.

## Out of scope

- New components (Sub-project 2).
- Theme file edits.
- Automatic rebuild of the Go shared library when icons change (contributors still run `bash lib/build.sh` as documented in `CLAUDE.md`).
- Cross-platform packaging of the generator (stdlib only; runs anywhere Python 3.10+ runs).
- Docs-page additions showing newly-extracted icons (none are new; `docs/shapes.rst` already lists the full set).
